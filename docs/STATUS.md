# Project status

A snapshot of what works today and what's next. Updated July 2026.

## Working

### Web editor (`web/`)
- Full authoring: 3D LED placement (shape presets + manual editing), gradient
  **and** image sources, six path types (line / sine / ellipse / spiral /
  polygon / free-drawn poly) with on-texture handle editing, focus mode with a
  magnifier loupe + color readout, per-track speed/chase, a speed-automation
  lane, animation-index assignment, editable timeline length/fps.
- One rasterized **source-texture** pipeline, so the loupe, timeline strip, and
  export agree pixel-for-pixel with the 3D preview.
- Bake to a fixed-rate raster (row = LED, column = frame).
- **Export** for RP2040: one-drag UF2 (MicroPython + player + pattern),
  MicroPython `.zip`, or raw `.leda`. Soft size/framerate warnings per device
  profile. The player ships **precompiled to `leda.mpy`** (armv6m / MicroPython
  v1.28) + a tiny loader `main.py`, so the board skips the ~3 s on-device compile
  every boot (measured 2.97 s → 1.22 s). Regenerate the blob with `npm run gen:mpy`
  after editing `rp2040.ts`; the build's `prebuild` hook fails on a stale blob.
- Project save/load (JSON), local library (IndexedDB), crash recovery, and a
  storage-version gate that clears a stale session after breaking changes.

### Device firmware (generated by the web export)
- MicroPython player for Pico / Pico W (WS2812 via PIO); template in
  `web/src/export/rp2040.ts`.
- Text control protocol over **BLE** (Nordic-UART-style) and **Wi‑Fi TCP :4550**;
  see [`protocol.md`](protocol.md).
- Wi‑Fi provisioning (credentials stored on-device) + UDP `LEDADISCOVER`
  discovery. Live settings: brightness, speed, solid color, data pin, name.

### Companion app (`iOS/` — iOS / iPadOS / macOS)
- Discover over BLE + Wi‑Fi (merged list / tiles), connect (Wi‑Fi-preferred, BLE
  fallback), control brightness/speed/solid/off/play, select/delete patterns,
  rename, set data pin, Wi‑Fi provisioning, info + live power, pattern upload
  over Wi‑Fi.

## Multi-device: incremental testing plan

The design (`multi-device-sync.md`) can be validated in stages, most of it before
a second Pico exists:

1. **Single Pico + phone** — everything except the wireless: per-device export
   (upload each device's slice, confirm it plays its subset), program numbering /
   selection, indexed-color/palette/dither format, the LED-0 indicator rendering,
   and the follower's *behavior* (phase-lock, apply program/brightness, teardown,
   reboot-search fallback) driven by GATT-written test sync packets.
2. **A Raspberry Pi (or Linux box) as a scriptable leader/sniffer** — Linux/BlueZ
   can emit real manufacturer-data beacons (iOS/macOS can't), so use it to prove
   the actual advertise/scan transport and to prototype the beacon protocol in
   CPython before porting to MicroPython. It can also scan to verify a real Pico
   leader's advert payload.
3. **Two real Pico W devices** — the only thing that truly needs a second board:
   end-to-end "do they hold visual sync over time," plus the seam-palette check
   once they're physically adjacent.

## Format sizing — key facts

The on-disk pattern format is [`file-format.md`](file-format.md) (LEDA v1). For
the export-size work, the numbers that matter:
- Raster size = `frames × LEDs × bytes-per-pixel`. The levers are **independent
  and multiply**: frame rate, per-pixel depth (RGB888→565→indexed-8), and
  splitting across N devices (divides the **spatial** dimension only — frame
  count stays the same).
- Example: 60 s × 2000 LEDs @ 30 fps RGB888 = 10.8 MB whole → 10 fps (÷3) →
  indexed-8 (÷3) → 8 devices (÷8) ≈ **~150 KB per device** (+ palette).
- Prefer **one shared palette per program** (not per device) so adjacent devices
  match exactly at a seam.
- The firmware's dumb fixed-stride array-stepping precludes on-device
  decompression/eval, which is *why* the levers are limited to fixed-stride
  formats; for genuinely huge shows the clean path is bigger flash / an SD card
  the firmware streams frames from, not compression.
