# LED Animator — device firmware (Rust)

A Rust reimplementation of the device firmware (the MicroPython player generated by
[`web/src/export/rp2040.ts`](../web/src/export/rp2040.ts)), on the RP2040 / Pico W
today and the RP2350 / Pico 2 W next. Its own crate for now; **merges in / replaces
the Python once it reaches parity.** Built on **embassy** (async), because the CYW43
Wi-Fi/BLE driver is embassy-native.

For the full plan, status table, Python→Rust mapping, and gotchas, read
**[`docs/PORT.md`](docs/PORT.md)**. This file is the orientation + workflow.

## How it's put together

`embassy_executor::main` sets everything up, then **becomes the player loop**; the
rest run as spawned async tasks that all share the one radio + executor:

- **player** (in `main.rs`) — reads a snapshot of `state::CONTROL` each frame and
  renders per mode (Play the selected `.leda` at speed% / Solid color / Off), on
  **PIO1 + DMA_CH2**.
- **`usb::usb_task`** — one USB device: the picotool **reset interface** + a CDC
  **serial logger** (`log::info!` → `/dev/cu.usbmodem*`).
- **`radio::runner_task`** — the CYW43 driver, on **PIO0 + DMA_CH0/CH1**.
- **`ble::ble_task`** — trouble-host NUS control service; a command written to RX is
  run through `protocol::dispatch` and the reply notifies on TX.

Shared state is small and explicit: `state::CONTROL` (an async `Mutex`) carries
mode/brightness/speed/solid/file/name between the control channels and the player.
`protocol::dispatch` is **transport-agnostic** — the same command handling backs
BLE now and Wi-Fi/TCP later.

**Boot flow** (`main`): USB up (so reflash always works) → clear strip →
`BootStage::Storage` (mount littlefs) → recovery counter bump → config load →
`BootStage::Radio` (CYW43 bring-up; a dead chip hangs here on frozen-blue LED 0) →
green "complete" flash → commit recovery counter → **player loop** (+ BLE task).

Module map: [`main.rs`] boot + player · [`ws2812.rs`] LED output · [`leda.rs`]
pattern format · [`state.rs`] shared control state · [`protocol.rs`] command
dispatch · [`ble.rs`] BLE transport · [`radio.rs`] CYW43 · [`fs.rs`] littlefs ·
[`recovery.rs`] power-cycle ritual · [`status.rs`] LED-0 diagnostics · [`usb.rs`]
dev tooling · [`panic.rs`] crash flutter.

## Dev workflow (no BOOTSEL button)

Once the Phase-C firmware is on the device, **all iteration is CLI** — the picotool
reset interface replaces the physical button.

```bash
cd firmware-rust
export PATH="$HOME/.rustup/toolchains/stable-aarch64-apple-darwin/bin:$PATH"

# 1. build
cargo build --release

# 2. package a UF2
~/.cargo/bin/elf2uf2-rs target/thumbv6m-none-eabi/release/led-animator-fw led-animator.uf2

# 3. flash — no button: drop to BOOTSEL over USB, then load
picotool reboot -f -u          # running firmware -> BOOTSEL (via the reset interface)
picotool load -x led-animator.uf2

# 4. watch the serial log
screen /dev/cu.usbmodemLEDA_00012 115200      # (Ctrl-A K to quit)
```

First-ever flash of a device with no reset interface yet still needs the physical
BOOTSEL button once; after that, step 3 is button-free.

Observability note: the CDC log only reaches the host once it re-enumerates (~2 s
after a reboot), so one-shot boot lines can be missed — the periodic `alive — frame
N` heartbeat is the reliable signal.

## Controlling it (BLE)

The device advertises the LED Animator **service UUID**
`4C454441-0001-4000-8000-4C4544415254` with its name in the scan response. Write a
text command to the **RX** characteristic `6E40…0002`; replies notify on **TX**
`6E40…0003` (see [`../docs/protocol.md`](../docs/protocol.md)).

Drive it from any machine with a free BLE central:

```bash
python3 -m venv venv && venv/bin/pip install bleak
venv/bin/python tools/ble_test.py     # scans, connects, runs PING/INFO/SOLID/BRIGHT/OFF…
```

or use nRF Connect (write `INFO` / `SOLID 0 255 0` to RX, read TX).
