# LED Animator

Design LED animations in the browser, flash them to a microcontroller, and
control them from your phone or Mac.

You author patterns in a **web editor** — place LEDs in 3D, sample
gradients/textures/images along per-track paths, and tune speed/chase over a
timeline. Everything **bakes down to a fixed-rate raster** (row = LED, column =
frame) that a device plays back with a dead-simple fixed-rate loop. The browser
preview steps that *same* raster, so what you see matches the hardware. A
**companion app** (iOS/macOS) then discovers devices over Bluetooth or Wi‑Fi and
controls playback, brightness, color, and Wi‑Fi setup.

## The pipeline

```
author (web editor)
   │  place LEDs · gradient/image sources · paths · speed/chase · timeline
   ▼
bake  → fixed-rate raster  (row = LED, column = frame, RGB)
   │
   ▼
export (web editor)
   ├─ One-drag UF2    → drop onto a blank Pico W (MicroPython + player + pattern)
   ├─ MicroPython .zip → for a Pico already running MicroPython
   └─ Pattern .leda    → just the raster, to sit next to an existing player
   ▼
flash  (drag the UF2 onto the RPI-RP2 drive)
   ▼
control (companion app)
   └─ discover over BLE / Wi-Fi · select pattern · brightness · speed · color · Wi-Fi setup
```

New to microcontrollers? Start with the **[Getting Started guide](docs/getting-started.md)**
— it walks flashing and connecting with no prior firmware experience.

The **device firmware** is a MicroPython player generated by the web app — its
source lives as templates in [`web/src/export/rp2040.ts`](web/src/export/rp2040.ts),
and export bundles it (plus your pattern and settings) into a ready-to-flash
image. There is no separate `firmware/` build to compile.

Project-wide files (this README, `LICENSE`, the GitHub Pages workflow in
`.github/`) live at the root.

## Quick start

**Editor** (from `web/`):

```bash
npm install
npm run dev      # http://localhost:5173
```

Or use the hosted GitHub Pages build. Author a pattern, then **Export → One-drag
UF2**, and follow the [Getting Started guide](docs/getting-started.md) to flash it.

**Companion app**: open the Xcode project under `iOS/LED Animator/` and run on an
iPhone/iPad or Mac. See [`iOS/README.md`](iOS/README.md).

## Data format & control protocol at a glance

The two contracts everything depends on — full specs in
[`docs/file-format.md`](docs/file-format.md) and [`docs/protocol.md`](docs/protocol.md).

**Pattern file (`.leda`):** a 20-byte header + frame-major RGB888 pixels. All
integers little-endian.

| Offset | Size | Field |
|---:|---:|---|
| 0 | 4 | magic `LEDA` |
| 4 | 1 | version (`1`) |
| 5 | 1 | pixel format (`0` = RGB888) |
| 6 | 2 | numLeds (uint16) |
| 8 | 4 | numFrames (uint32) |
| 12 | 2 | fps (uint16) |
| 14 | 1 | sync role (`0` standalone … `4` auto-elect) |
| 15 | 1 | group id (uint8) |
| 16 | 1 | device / render-slice id (uint8) |
| 17 | 1 | sync flags (on-loss policy + startup) |
| 18 | 2 | reserved |
| 20 | … | `numFrames × numLeds × (R,G,B)` |

The sync role/group/device/flags carry the multi-device configuration — a plain
single-device pattern is all-zero there. See [`file-format.md`](docs/file-format.md).

**Control protocol:** one-line text commands (e.g. `INFO`, `SELECT <name>`,
`BRIGHT 50`), identical over two transports:
- **BLE** — a custom service UUID + Nordic-UART RX/TX characteristics.
- **Wi‑Fi** — the same commands over **TCP port 4550**; discovery via a UDP
  `LEDADISCOVER` broadcast (also 4550).

## Hardware supported today

- **Raspberry Pi Pico / Pico W (RP2040)** running MicroPython, driving WS2812
  ("NeoPixel") strips via PIO. The Pico **W** adds BLE + Wi‑Fi control.

ATmega328 (Arduino) and Raspberry Pi targets are profiled in the exporter (their
size limits are shown) but their export paths are planned, not built.
